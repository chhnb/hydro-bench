# Taichi vs Native CUDA Comprehensive Alignment Validation Plan

## Goal Description

Establish an end-to-end alignment validation pipeline between the Taichi and native CUDA hydrodynamics implementations across all four cases (F1_6.7K, F1_207K, F2_24K, F2_207K) and both precisions (fp32, fp64). The pipeline verifies alignment along three independent dimensions:

1. **High-precision binary state dumps** — per-cell H/U/V/Z/W and per-edge F0..F3, with distribution statistics and worst-cell localization
2. **Native simulation output files** — H2U2V2.OUT / ZUV.OUT / SIDE.OUT / XY-TEC.DAT / TIMELOG.OUT generated by both sides at matching cadence, compared as the user-facing alignment ground truth
3. **Physical conservation quantities** — mass, momentum, energy, and boundary inflow integrals, which are insensitive to chaotic floating-point divergence

Validation is performed at a fixed set of step checkpoints `{1, 100, 900, 7200, 36000, 72000}` (capped at 72,000 to keep full-matrix runtime bounded). The plan includes fixing one identified algorithmic mismatch (F1_207K_fp32 step=1 not bit-exact) and producing a single JSON+Markdown report covering all 56 (case × precision × step) combinations.

The success of this pipeline establishes that the Taichi implementation is functionally equivalent to native CUDA for all production use cases, with explicit, measurable tolerance bounds appropriate to each precision and step regime.

## Acceptance Criteria

Following TDD philosophy, each criterion includes positive and negative tests for deterministic verification.

- AC-1: F1_207K_fp32 step=1 reaches algorithmic equivalence with native CUDA (sub-ulp match).
  - Positive Tests (expected to PASS):
    - After Stage 1 fix, running F1_207K_fp32 for step=1 produces H/U/V/Z max_abs diff < 1e-6 against native.
    - At step=1, ≥99% of cells are bit-exact (zero diff) for H/U/V/Z fields.
    - Worst-cell localization at step=1 reports cells whose KLAS values include 1 and 10 (the boundary types previously diverging).
  - Negative Tests (expected to FAIL):
    - Reverting the fix and rerunning yields step=1 H max_abs > 1e-5 (regression detected).
    - Injecting an off-by-one in the KLAS=1 iteration index causes step=1 to fail the ≥99% bit-exact threshold.

- AC-2: `scripts/check_correctness.py` supports a configurable step-checkpoint list, computes per-field distribution statistics, and emits a structured JSON report.
  - AC-2.1: A `--steps` flag accepts a comma-separated checkpoint list and runs each checkpoint independently, restoring initial state between checkpoints.
    - Positive: `python check_correctness.py F2_207K_fp32 --steps 1,100,900` produces three independent comparisons in one invocation.
    - Negative: Omitting the flag falls back to the existing single-step behavior without breaking any current call site.
  - AC-2.2: For every (H, U, V, Z, W, F0, F1, F2, F3) field, the report includes `max_abs`, `mean_abs`, `p50`, `p90`, `p99`, `p99.9`, threshold counts at `{1e-7, 1e-5, 1e-3, 1e-1}`, and the top-3 worst-cell entries with `(idx, native_val, taichi_val, klas, neighbor_klas)`.
    - Positive: Running on F2_207K_fp32 step=900 emits a JSON file containing all 9 field blocks with all listed sub-fields populated.
    - Negative: Removing the W field from output causes a schema validator to reject the report.
  - AC-2.3: The report is written to `results/alignment/{case}_{precision}_{step}.json` and a per-case Markdown summary is appended to `results/alignment/SUMMARY.md`.
    - Positive: After running 7 step checkpoints for F2_207K_fp32, exactly 7 JSON files exist with parseable content, and SUMMARY.md contains a row for each.
    - Negative: A run that fails halfway leaves earlier completed JSONs intact (idempotent).

- AC-3: Native CUDA benchmark binary supports a `--with-output` mode that calls `MeshData::outputToFile(jt, kt)` at the configured cadence, with NTOUTPUT overridable from the command line.
  - AC-3.1: Adding `--with-output` to the benchmark invocation produces non-empty H2U2V2.OUT, ZUV.OUT, SIDE.OUT, XY-TEC.DAT, and TIMELOG.OUT in the case's `OUTPUT/` directory.
    - Positive: Running F2_24K with `--with-output --ntoutput 1 --steps 900` writes a non-zero-byte H2U2V2.OUT containing exactly one (jt=0, kt=1) initial frame plus one frame at step 900.
    - Negative: Running without `--with-output` leaves all .OUT files at zero bytes (current behavior preserved).
  - AC-3.2: A `--ntoutput N` flag overrides the value parsed from TIME.DAT.
    - Positive: F2_207K with `--ntoutput 1 --steps 7200` writes 2 frames (initial + after 1 day), independent of TIME.DAT's NTOUTPUT=50.
    - Negative: Omitting `--ntoutput` falls back to TIME.DAT's value.
  - AC-3.3: The .OUT files preserve the exact format produced by the legacy `MeshData::outputToFile` (same `setprecision`, `setw`, newline rules, headers).
    - Positive: A bit-by-bit comparison of a regenerated H2U2V2.OUT against a reference frame produced by the legacy code path matches.
    - Negative: Changing `std::setprecision(4)` to `std::setprecision(6)` in the writer causes the comparison to flag every line as different.

- AC-4: Taichi side writes equivalent .OUT files via a new `taichi_impl/output_writer.py`, byte-format-compatible with the native writer for the deterministic precision/format rules.
  - AC-4.1: After running a case with output enabled, a Taichi-side `OUTPUT/` directory contains H2U2V2.OUT, ZUV.OUT, SIDE.OUT, XY-TEC.DAT, TIMELOG.OUT.
    - Positive: F2_24K_fp64 with output enabled writes all 5 files, and the line count of H2U2V2.OUT matches the native counterpart for the same step list.
    - Negative: Omitting any field (e.g., the W2 block in ZUV.OUT) causes the file-comparison stage to report a structural mismatch.
  - AC-4.2: Floating-point values in the Taichi-written files use `f"{val:10.4f}"` format with the same column widths and newline cadence as the native writer.
    - Positive: A line-by-line byte diff of the Taichi vs native H2U2V2.OUT for an fp64 case at step=1 reports zero differing lines.
    - Negative: Using `f"{val:.4e}"` format causes the comparison to flag every numeric line.
  - AC-4.3: SIDE.OUT (one-time geometry dump) reproduces the native side's default-precision streaming format for COSF/SINF/SIDE/AREA blocks.
    - Positive: A produced SIDE.OUT can be parsed by the same tokenizer that parses the native counterpart, and the parsed numeric values match within the format's representable precision.
    - Negative: Substituting fixed-precision formatting causes parser-rejected lines.

- AC-5: A conservation-quantity calculator computes mass, momentum (X/Y), kinetic energy, potential energy, and boundary inflow (KLAS=10 and KLAS=1) for any post-step state.
  - AC-5.1: For an fp64 case at any tested checkpoint, all 7 conservation quantities computed from native and Taichi states satisfy `|Δq / q| < 1e-12`.
    - Positive: F2_207K_fp64 at step=72000 reports all 7 relative diffs below 1e-12.
    - Negative: Injecting a 1e-10 perturbation in one Taichi cell's H value at step=72000 causes the mass conservation check to fail.
  - AC-5.2: For an fp32 case at step ≥ 100, all 7 conservation quantities satisfy `|Δq / q| < 1e-5`.
    - Positive: F2_207K_fp32 at step=900 and step=7200 report all 7 relative diffs below 1e-5.
    - Negative: Replacing one boundary cell's QT value with a 1% perturbation causes the KLAS=10 inflow check to fail.
  - AC-5.3: The conservation block is included in every JSON report alongside per-field statistics.
    - Positive: The JSON for F2_207K_fp32 step=7200 contains a `conservation` object with all 7 named entries.
    - Negative: Omitting the BC inflow entries causes a schema check to fail.

- AC-6: A new `scripts/compare_output_files.py` compares two `OUTPUT/` directories (native + taichi) producing both textual diff and parsed numeric diff statistics.
  - AC-6.1: Given two identical OUTPUT directories, the comparator reports zero differing lines and `text_match: true` for all 5 files.
    - Positive: Symlinking native OUTPUT to itself and running comparator yields a clean report.
    - Negative: Modifying one byte in H2U2V2.OUT produces a non-zero `diff_lines` count.
  - AC-6.2: For numeric blocks, the comparator parses H/U/V/Z/W values and reports `max_h_diff`, `max_u_diff`, `max_v_diff`, `max_z_diff`, `max_w_diff` per frame.
    - Positive: A run on F2_24K_fp32 step=900 produces a numeric report with all five `max_*_diff` keys present.
    - Negative: Producing a syntactically valid but semantically scrambled H2U2V2.OUT (wrong cell ordering) is detected as a numeric mismatch even if line count matches.

- AC-7: A driver script `scripts/run_alignment_full.sh` runs the entire matrix (4 cases × 2 precisions × 7 step checkpoints, capped at 72,000) and produces the consolidated `SUMMARY.md` and per-run JSON reports.
  - Positive: A single invocation produces 56 JSON files and one SUMMARY.md table containing one row per (case, precision, step), with PASS/FAIL verdict per row.
  - Negative: A failure in one case does not prevent the remaining cases from running and reporting independently.

- AC-8: All fp64 (case × step) entries in the matrix achieve PASS verdicts.
  - Positive: For every fp64 row in SUMMARY.md, the binary-dump field max_abs is below 1e-9, p99 below 1e-11, all 7 conservation relative diffs below 1e-12, and OUTPUT files are 100% line-identical.
  - Negative: A regression introducing a 1e-8 ulp drift in any fp64 path causes at least one fp64 row to flip to FAIL.

- AC-9: All fp32 (case × step=1) entries achieve PASS verdicts (algorithmic equivalence).
  - Positive: For every fp32 row at step=1 in SUMMARY.md, ≥99% cells bit-exact, max_abs < 1e-6 for state fields, max_abs < 1e-5 for fluxes, and OUTPUT files are 100% line-identical.
  - Negative: Reverting the F1_207K_fp32 fix causes the F1_207K_fp32 step=1 row to flip to FAIL.

- AC-10: All fp32 (case × step ≥ 100) entries achieve PASS verdicts under hard thresholds.
  - Positive: For every fp32 row at step in {100, 900, 7200, 36000, 72000} in SUMMARY.md: all 7 conservation relative diffs < 1e-5, p99 diff per state field < 1e-3, OUTPUT files ≥ 99% lines identical with H deviations < 0.01 on differing lines, and zero NaN/Inf cells.
  - Negative: Introducing a path that randomly drops 0.1% of cell updates causes the conservation mass check to fail.

## Path Boundaries

Path boundaries define the acceptable range of implementation quality and choices.

### Upper Bound (Maximum Acceptable Scope)

The implementation includes: a step-callback hook in all four Taichi `step_fn` closures; an enhanced `check_correctness.py` with full distribution statistics, worst-cell localization, conservation calculator, and JSON+Markdown emission; native CUDA `--with-output --ntoutput N` mode in both the fp64 and fp32 build variants; a Taichi `output_writer.py` reproducing all five .OUT formats; a `compare_output_files.py` doing both textual and numeric comparison; a `run_alignment_full.sh` driver that walks the entire 56-cell matrix; a JSON schema for the report; the F1_207K_fp32 step=1 algorithmic fix; and a brief drift-curve plotting script for the F2_207K_fp32 long-step case. All 10 acceptance criteria are independently testable and the test inputs are checked into the repo.

### Lower Bound (Minimum Acceptable Scope)

The implementation includes: a step-callback hook on `step_fn` for all four Taichi entry points; multi-step support and conservation quantities in `check_correctness.py` with JSON output; native `--with-output` (NTOUTPUT override may be hardcoded to 1); a Taichi `output_writer.py` for H2U2V2.OUT and ZUV.OUT (the two highest-value diff targets); `compare_output_files.py` with at least textual diff plus numeric diff for H/U/V/Z; a `run_alignment_full.sh` driver covering all 56 cells; the F1_207K_fp32 step=1 fix; and SUMMARY.md emission. SIDE.OUT, XY-TEC.DAT, TIMELOG.OUT comparison may be limited to existence-only checks. The drift-curve plotting script is omitted.

### Allowed Choices

- Can use: Python 3 for harness and comparison code; numpy/struct for binary I/O; subprocess for invoking Taichi in fresh interpreters (preserving the existing pattern); standard `difflib` for textual diff; CUDA `cuda-toolkit/bin/nvcc` already in PATH for rebuilding native; a callback parameter on `step_fn(on_step=fn)` to support intermediate dumps (per user decision); NTOUTPUT=1 override for alignment runs while preserving original NTOUTPUT for benchmark runs (per user decision); a 72,000-step cap on the alignment matrix (per user decision); hard thresholds on conservation and per-field statistics (per user decision).
- Cannot use: tools or libraries not already declared in `requirements.txt` without first updating it; modifications to Taichi runtime source; removal of the existing single-shot dump mode; relaxing the fp64 thresholds below the values specified in the draft.

### Conceptual Approach

A high-level walkthrough of one viable implementation:

```
F1_207K_fp32 step=1 fix:
  1. Add a probe kernel that dumps (H, U, V, Z, F0..F3) for cells whose
     KLAS edge ∈ {1, 10} after step=1 from both sides.
  2. Diff the probe outputs cell-by-cell, identify the first divergent cell
     and edge.
  3. Inspect the BC dispatch path for that KLAS in
     taichi_impl/F1_hydro_taichi_2kernel_fp32.py and
     cuda_native_impl/hydro-cal-src/src/functors.cu (CalculateKlas1 /
     CalculateKlas10).
  4. Bring the Taichi iteration's break condition / FTZ behavior into line
     with native by inserting fp64 helpers (already an established pattern
     in the file).

step_fn callback hook:
  step_fn(on_step=lambda s: ...)
    where on_step is invoked after sync at step s == checkpoint, so the
    caller can read H/U/V/Z/F0..F3 to_numpy() and persist.
  Existing call sites that don't pass on_step keep the all-at-once
  behavior (default callback is None).

Native --with-output:
  Add CLI parsing for --with-output and --ntoutput in benchmark.cu.
  Inside runSteps(), call mesh_view.ToHost(mesh_data) +
  mesh_data.outputToFile(jt, kt) at the cadence dictated by NTOUTPUT.
  The MeshData ofstream members are already opened by the constructor.

Taichi output_writer:
  output_writer.write_h2u2v2(path, jt, kt, dt, h, u, v) reproduces the
  exact native format: header line "JT=... KT=... DT=...", then "H2="
  marker followed by 10-per-line, 100-per-block fixed-precision values.
  Same for ZUV.OUT (Z, W, FI) and SIDE.OUT (one-time, default-precision).

compare_output_files:
  text-mode: difflib.unified_diff line by line.
  numeric-mode: a line-by-line tokenizer that detects "H2=" / "U2=" /
  "V2=" / "Z2=" / "W2=" markers and parses N floats into numpy arrays
  for diff statistics.

run_alignment_full.sh:
  for case in F1_6.7K F2_24K F1_207K F2_207K; do
    for prec in fp32 fp64; do
      for step in 1 100 900 7200 36000 72000; do
        python scripts/check_correctness.py "${case}_${prec}" --steps "$step"
      done
    done
  done
  Then aggregate JSON files into SUMMARY.md.
```

- Per user decision, fp32 step ≥ 100 thresholds (conservation 1e-5, p99 1e-3, OUTPUT ≥ 99% line match) are hard requirements that gate PASS verdicts in SUMMARY.md.

- `SIDE.OUT` is emitted once at `(jt=0, kt=1)` using default streaming precision. This block is treated as a fixed reference; the Taichi writer must reproduce the same default-precision streaming format.

## 1. 验证维度

### 1.1 高精度二进制 dump（已有，将增强）

| 字段 | 维度 | 说明 |
|------|------|------|
| H, U, V, Z, W | CELL | 单元格状态（W=√(U²+V²) 派生） |
| F0, F1, F2, F3 | 4·CELL | 边通量 |

**升级点**：补 W；多 step 检查点；分布统计（mean/p50/p90/p99/p99.9）；worst-K cells 定位。

### 1.2 仿真原生输出文件（新增）

`mesh.cpp::outputToFile(jt, kt)` 在每 NTOUTPUT 天写：

| 文件 | 内容 | 频率 | 精度 |
|------|------|------|------|
| **SIDE.OUT** | COSF, SINF, SIDE, AREA | 一次（jt=0, kt=1） | 默认 | 
| **H2U2V2.OUT** | H2, U2, V2 + JT/KT 头 | 每 NTOUTPUT 天 | `setprecision(4)` |
| **ZUV.OUT** | Z2, W2, FI（流向角） | 每 NTOUTPUT 天 | `setprecision(4)` |
| **XY-TEC.DAT** | TEC 可视化（X,Y,H2,Z2,U2,V2,W2 + 单元连接） | 每 NTOUTPUT 天 | `setprecision(4)` |
| **TIMELOG.OUT** | jt/NDAYS 进度比 | 每 NTOUTPUT 天 | `setprecision(4)` |

**重要**：当前 `benchmark.cu` 没有调用 `outputToFile()`（OUTPUT 目录下的 *.OUT 都是 0 字节）。需要：
1. native 侧加 `--with-output` 选项，按 NTOUTPUT 天调用 `outputToFile()`。
2. Taichi 侧实现等价的 `dump_output_files()`，相同格式（4 位小数）。
3. 写一个 `compare_output_files.py` 做文本/数值双重对比。

**为什么测这个**：
- 这些文件是真实下游用户消费的格式 → 是产品级对齐的金标准
- `setprecision(4)` 自带容忍 ~1e-4 → fp32 长步漂移天然落进容忍区
- 文件按 NTOUTPUT 多日 dump → 自带 step-axis 检查点
- TEC 可以直接做可视化 diff

### 1.3 守恒量（新增，混沌不敏感）

每个检查点计算：

| 量 | 公式 | 物理含义 |
|----|------|----------|
| 总水量 | Σ H·AREA | 质量守恒（除 BC 影响） |
| 动量 X | Σ H·U·AREA | x 方向动量 |
| 动量 Y | Σ H·V·AREA | y 方向动量 |
| 动能 | Σ ½H(U²+V²)·AREA | 总动能 |
| 位能 | Σ ½g·H²·AREA | 总位能 |
| KLAS=10 累计入流 | Σ F0·SIDE on KLAS=10 | BC 通量积分 |
| KLAS=1 累计入流 | Σ F0·SIDE on KLAS=1 | 水位 BC 通量 |
| H 极值 | (H_min, H_max) | 数值稳定性指示 |
| NaN/Inf 计数 | finite cells / total | 健康检查 |

混沌发散不影响守恒量 — 这是 fp32 长步对齐的核心判据。

## 附录 A：fp32 PTX 指令模式差异（已实测）

| 指令 | Taichi (默认) | nvcc `--fmad=false` | nvcc `--ftz=true --prec-div=false` |
|------|---------------|---------------------|------------------------------------|
| mul fp32 | `mul.ftz.f32` | `FMUL`（非 FTZ） | `FMUL.FTZ` |
| add fp32 | `add.ftz.f32` | `FADD`（非 FTZ） | `FADD.FTZ` |
| FMA 合并 | `fma.rn.ftz.f32`（LLVM 仍合并）| `FFMA`（仍出现，来自 fma() 调用） | `FFMA` |
| div fp32 | `div.approx.ftz.f32` | `div.rn.f32`（IEEE） | `div.approx.f32` |
| sqrt fp32 | `sqrt.rn.ftz.f32` | `sqrt.rn.f32` | `sqrt.rn.ftz.f32` |

**结论**：用 `nvcc --fmad=false --ftz=true --prec-div=false --prec-sqrt=true` 可以让 native 更接近 Taichi 的 PTX 模式，但**不会减小 fp32 长步漂移**（实测：换 flag 后 native 自比都漂 0.79）。这只在 step=1 bit-exact 调试有用。
