E2E Launch Path Optimization

Taichi + CUDA Graph：E2E Launch 优化

这一层优化不改变 kernel 算法，而是把 Taichi step_fn 捕获成 CUDA Graph 后 replay。

03 / 05

我们怎么做

  • patch Taichi runtime，暴露 CUDA stream 与 capture 控制接口。
  • 将 Taichi launch 路由到用户 stream：cuda_set_stream()。
  • capture 期间跳过不允许的 sync：cuda_set_capture_active()。
  • 使用 bump allocator pool，避免 capture 中的 cuMemAllocAsync。
  • 复用 persistent arg/result buffers，减少每次 launch 的分配和 H2D 参数拷贝。
  • 小 mesh 支持 multi-step capture，一个 graph launch 执行多个 timestep。
Vanilla Taichi 直接 stream capture 会遇到 CUDA_ERROR_STREAM_CAPTURE_IMPLICIT；patch 后可以从 Python 捕获并 replay Taichi kernel。

Agent-Based CUDA Optimization

Aker Agent：bit-exact 进化优化

Aker 把 CUDA 优化组织成受约束的 evolutionary loop：生成候选、自动验证、benchmark 排名，再把有效节点写回 leaderboard。

05 / 05

进化迭代机制

1
Spec 锁定
固定 case、父节点、可改文件、性能指标和 exact correctness gate。
2
Agent 变异
读取 leaderboard、guidance、SASS/NCU 证据，在新 node 写入 kernel.cu。
3
Reviewer 筛选
阻止越界改动、缺失节点、明显改变语义或无法解释的风险 patch。
4
自动评测
编译、跑 1/10/100/899/7199 步，对 H/U/V/Z/W/F0-F3 exact diff。
5
排序继承
通过者 benchmark，写 leaderboard；下一轮从 best 和成功分支继续变异。
这不是单次代码生成，而是 generate → review → build → validate → benchmark → rank 的闭环搜索；并行 slots 只是同时探索多个候选节点。

技术边界

  • Exact track 只接受不改变 native v0 数值轨迹的局部 CUDA 优化。
  • 失败节点包括 compile fail、accuracy diff、reviewer 拒绝或 agent 没产出 node。
  • 更大范围 Flux rewrite 要单独开 experimental track，并定义 relaxed validation。
  • 下一步让 agent 基于 NCU dynamic metrics 定向减少 FP64、branch 和 predicate work。
