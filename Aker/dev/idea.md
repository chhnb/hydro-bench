# AI-driven graph growth 的几个核心设计原则

适用于一类任务：**让 AI 迭代地生长一张候选解的图，每个节点被评估后再产出下一个**。

问题的基本形状：
- 每一轮，AI 基于当前图的状态决定下一个要做什么
- 它写代码 / 文档 / 方案，系统评估结果
- 成功的进入 leaderboard，失败的留在图里，都是下一轮的输入
- 循环

这个形状下，最关键的几条设计决定。

---

## 1. 图就是记忆

AI 每一轮不是重新开始思考，而是读前几轮积累下来的图。**图上承载的信息决定了 AI 能不能积累智慧**。

- 每个节点除了产出物本身（代码），还要有结构化元数据：对父节点做了什么变化、为什么这么做、用了什么技术
- 单独一份设计文档描述"这个节点现在是什么样"——与"它相对父节点改了什么"是两个问题，用两份文档分开
- 失败节点不删，当负样本留着——AI 下一轮看到"这方向已经试过且失败"就不会重做

没这一套，AI 每一轮只能重读所有代码重新推理。信息密度太低，记忆等于没有。

---

## 2. 格式只用在控制流

LLM 的回答服务于两种用途：**被读**（人或另一个 LLM）、**被 parse**（程序据此分支）。

第二种只需要**一个**硬格式 token 就够——比如 "结尾必须出现 `VERDICT: PASS/RETRY`"，程序 grep 这一行决定继续还是停。

除此之外所有内容——评审理由、设计理由、决策解释——完全自由。把"我想 parse 的需求"伪装成"LLM 必须遵守的推理格式"是一个常见的坑：每加一条格式规则就限制一点 LLM 的表达空间，叠起来就是限制它的思考空间。

数据字段（JSON 里的 parents / techniques / status）不属于这个范畴——那是文件格式契约，不是推理格式契约。AI 往文件写 JSON 和 AI 用固定开头段落写评审是两件事。

---

## 3. prompt 要反 anchoring

列"可能的优化方向清单"会让 AI 机械对号入座，搜索空间被清单本身压死。prompt 里每出现一个 artifact——baseline / reference / 方向清单 / 固定参数 / 早期 profiling 数据——都在把 AI 往这个 artifact 的方向推。

解法是**把 prompt 从"答案清单"改成"诊断问题"**：不是"你可以试试这些方向"，而是"当前最快方案的时间花在哪？先定位瓶颈再攻"。AI 被迫从第一性原理推理，而不是从例子类比。

同一类决策的其他表达：
- spec 只说**要做什么**、不说**怎么做**
- 参考实现 / baseline 默认不存在（避免"正确答案"锚定搜索）
- 重型工具（profiler / microarch 级分析）是 late-stage——早用会把搜索锁死在起始点的瓶颈上

---

## 4. 同一大脑演对抗对

很多 AI 系统用"一个 agent 做事、另一个 agent 审查"的对抗对。但两个角色不必是两个模型——同一个 LLM 底座能胜任，靠三件事差异化：

- **不同 prompt**：审查角色的"不许管什么"列表比"要管什么"列表更长——范围蔓延是审查者的主要失败模式
- **非对称 sandbox**：做事的有读写执行权限、审查的只读——**物理上**保证审查者只能判不能改
- **session 策略**：审查角色每轮重开，长 session 看多了 PASS 会松懈

sandbox 是这套里最 load-bearing 的。"不许编辑"作为 prompt 指令会被违反；作为 OS 权限不会。

---

## 5. 记忆长度是 trade-off

AI 跨轮保留 session（记忆）有两个失败模式：

- **不保留** → 每轮从零开始，丢掉上轮的推理肌肉
- **永久保留** → 陷进某个错误 mental model 出不来（sunk cost thinking，attention drift）

都不好。做法是**让每个新 session 抽一个随机寿命**（比如 uniform `[1, 5]` 轮）。两种失败模式都被转成概率事件，避开任何一种结构性失败。

---

**一句话总结**：

图承载推理痕迹；格式只用于控制信号；prompt 反 anchoring；对抗对靠 sandbox 不靠措辞；记忆长度要随机不要固定。
